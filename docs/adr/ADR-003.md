# ADR-003: Agent Skill Computation Strategy -- Intersection

| Field      | Value              |
|------------|--------------------|
| Status     | Accepted           |
| Date       | 2026-02-17         |
| Deciders   | Jeremy (maintainer) |

## Context

In the `create-agent-system`, each agent can be assigned **skills** (reusable capability modules like `scoring`, `tdd-workflow`, `visual-qa`, etc.). Two independent data sources determine which skills exist:

1. **`AGENT_DEFAULT_SKILLS`** -- A static mapping that defines which skills each agent is *designed to use*. For example, `frontend-dev` is designed to use `scoring`, `tdd-workflow`, `visual-qa`, and `design-system`, but has no use for `ticket-writing` or `adr-writing`.

2. **`preset.skills`** -- Each preset (solo-dev, small-team, full-team) activates a subset of all available skills. For example, `solo-dev` activates `scoring`, `tdd-workflow`, `ticket-writing`, and `cr-process` -- but not `visual-qa`, `design-system`, or `adr-writing`.

The question: **When rendering an agent's template, which skills should appear in its `skills` frontmatter field?**

This matters because:
- Skills listed in the frontmatter determine what the agent references during execution.
- Giving an agent a skill it does not know how to use creates noise and confusion.
- Omitting a skill that the preset has activated (but the agent should not use) is correct behavior, not a bug.

### Concrete Example

Given:
```
AGENT_DEFAULT_SKILLS['frontend-dev'] = ['scoring', 'tdd-workflow', 'visual-qa', 'design-system']
preset('solo-dev').skills = ['scoring', 'tdd-workflow', 'ticket-writing', 'cr-process']
```

What should `frontend-dev` get in the `solo-dev` preset?

| Strategy     | Result                                                    |
|--------------|-----------------------------------------------------------|
| Intersection | `['scoring', 'tdd-workflow']`                             |
| Union        | `['scoring', 'tdd-workflow', 'visual-qa', 'design-system', 'ticket-writing', 'cr-process']` |
| Preset only  | `['scoring', 'tdd-workflow', 'ticket-writing', 'cr-process']` |

## Decision

Use **intersection**: an agent receives only skills that are **both** in its default skill set **and** activated by the preset.

```typescript
agentSkills = AGENT_DEFAULT_SKILLS[agent].filter(skill => preset.skills.includes(skill))
```

Or expressed as set notation:

```
agentSkills(agent, preset) = AGENT_DEFAULT_SKILLS[agent] ∩ preset.skills
```

This is implemented in the template renderer (`src/core/template-renderer.ts`) when computing the `skills` frontmatter value for each agent template.

### AGENT_DEFAULT_SKILLS Reference

```typescript
// src/utils/constants.ts
export const AGENT_DEFAULT_SKILLS: Record<AgentName, SkillName[]> = {
  'po-pm':        ['scoring', 'ticket-writing'],
  'architect':    ['scoring', 'adr-writing'],
  'cto':          ['scoring', 'cr-process'],
  'designer':     ['scoring', 'visual-qa', 'design-system'],
  'test-writer':  ['scoring', 'tdd-workflow'],
  'frontend-dev': ['scoring', 'tdd-workflow', 'visual-qa', 'design-system'],
  'backend-dev':  ['scoring', 'tdd-workflow'],
  'qa-reviewer':  ['scoring', 'visual-qa', 'cr-process'],
};
```

### Example Outputs

| Agent        | solo-dev preset skills                 | small-team preset skills                              |
|--------------|----------------------------------------|-------------------------------------------------------|
| po-pm        | `scoring, ticket-writing`              | `scoring, ticket-writing`                             |
| architect    | *(not enabled in solo-dev)*            | `scoring, adr-writing`                                |
| cto          | *(not enabled in solo-dev)*            | `scoring, cr-process`                                 |
| designer     | *(not enabled in solo-dev)*            | `scoring, visual-qa, design-system`                   |
| test-writer  | `scoring, tdd-workflow`                | `scoring, tdd-workflow`                               |
| frontend-dev | `scoring, tdd-workflow`                | `scoring, tdd-workflow, visual-qa, design-system`     |
| backend-dev  | `scoring, tdd-workflow`                | `scoring, tdd-workflow`                               |
| qa-reviewer  | `scoring, cr-process`                  | `scoring, visual-qa, cr-process`                      |

Note how `frontend-dev` in `solo-dev` loses `visual-qa` and `design-system` (because the solo-dev preset does not activate those skills), while in `small-team` it gains them back. This is the desired behavior: solo developers do not need visual QA overhead, but small teams do.

## Alternatives Considered

| Criterion                   | Intersection (chosen)                      | Union                                          | Preset only                                    | Manual mapping per preset                   |
|-----------------------------|--------------------------------------------|-------------------------------------------------|------------------------------------------------|---------------------------------------------|
| **Formula**                 | `defaults[agent] ∩ preset.skills`          | `defaults[agent] ∪ preset.skills`               | `preset.skills` (ignore defaults)              | `presetAgentSkills[preset][agent]`          |
| **Agent gets irrelevant skills?** | No                                   | Yes -- agent gets skills it does not know how to use | Yes -- e.g., `backend-dev` gets `visual-qa` | No (if correctly configured)                |
| **Preset can restrict?**    | Yes (its core purpose)                     | No (union only adds, never removes)             | N/A (defaults are ignored)                     | Yes                                         |
| **Maintenance burden**      | Low -- two simple lists                    | Low -- but produces bad results                 | Low -- but produces bad results                | High -- N presets x M agents combinations   |
| **Adding a new agent**      | Add one entry to `AGENT_DEFAULT_SKILLS`    | Same                                            | Nothing needed (all get all)                   | Add entry to every preset                   |
| **Adding a new skill**      | Add to relevant agents' defaults + relevant presets | Same                                  | Add to relevant presets                        | Add to every preset-agent combination       |
| **Predictability**          | High -- developer can reason about result from two inputs | Medium -- union can surprise     | Low -- agents get unexpected skills            | High -- but only if all combinations are correct |

### Why not union?

Union (`AGENT_DEFAULT_SKILLS[agent] ∪ preset.skills`) gives every agent every skill from both sources. This means `backend-dev` would receive `visual-qa` (from the preset) even though it has no visual QA responsibilities. The `po-pm` agent would receive `tdd-workflow` even though it writes tickets, not tests. This creates noise in agent prompts and can lead to confused behavior -- an agent might try to follow a skill protocol that is irrelevant to its role.

### Why not preset only?

Using only `preset.skills` (ignoring `AGENT_DEFAULT_SKILLS` entirely) means every enabled agent in a preset gets the same skill set. The `backend-dev` gets `ticket-writing`, the `po-pm` gets `tdd-workflow`, and the `designer` gets `cr-process`. This violates the single-responsibility principle at the agent level. Each agent is specialized; its skills should reflect that specialization.

### Why not manual mapping per preset?

A manual mapping (e.g., `solo-dev.agents.frontend-dev.skills = ['scoring', 'tdd-workflow']`) gives maximum control but introduces combinatorial maintenance burden. With 8 agents and 3 presets, that is 24 skill lists to maintain. When a 4th preset is added, 8 more lists are needed. When a new skill is introduced, every relevant preset-agent pair must be updated. This is exactly the kind of duplication that leads to drift and bugs. The intersection approach derives the correct answer from two single-source-of-truth lists.

## Consequences

### Positive

- **Principle of least surprise** -- An agent only gets skills it is designed to use. A preset can deactivate skills globally, but it cannot give an agent a skill it was never meant to have. The result is always a subset of what the agent knows.
- **Simple mental model** -- Developers can answer "what skills does agent X get in preset Y?" by looking at two short lists and taking their overlap. No need to trace through complex configuration merging.
- **Easy to extend** -- Adding a new preset requires only listing which skills it activates. The intersection logic automatically computes the correct per-agent assignments. Adding a new agent requires only listing its default skills.
- **Single source of truth** -- Agent capabilities are defined once in `AGENT_DEFAULT_SKILLS`. Preset scope is defined once in `preset.skills`. There is no duplication.
- **Validates spec correctness** -- The intersection approach naturally catches spec errors. If a preset activates a skill but no agent defaults include it, that skill is effectively unused -- which can be flagged as a warning during validation.

### Negative

- **Implicit behavior** -- The intersection is computed at render time, not declared explicitly. A developer looking at a preset file alone cannot see exactly which skills each agent will receive; they need to also know `AGENT_DEFAULT_SKILLS`. This is mitigated by (a) documenting `AGENT_DEFAULT_SKILLS` clearly in constants, and (b) the `validate` subcommand can print the computed skill assignments.
- **Cannot give an agent a "bonus" skill per-preset** -- If a specific preset needs `frontend-dev` to have `adr-writing` (not in its defaults), the intersection approach cannot express this without modifying `AGENT_DEFAULT_SKILLS`. This is intentional -- if `frontend-dev` needs `adr-writing`, it should be added to its defaults globally, not as a per-preset override. If per-preset overrides become necessary in the future (v0.2.0 Custom preset), this can be addressed by adding an optional `skillOverrides` field to the preset schema.
- **Empty skill list edge case** -- If a preset activates none of an agent's default skills, the agent gets an empty skill list. This is valid (the agent operates without skills) but might surprise users. The `validate` subcommand should warn when an enabled agent has zero skills.
