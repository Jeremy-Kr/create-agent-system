# ADR-002: Template Resource Path Strategy -- import.meta.url

| Field      | Value              |
|------------|--------------------|
| Status     | Accepted           |
| Date       | 2026-02-17         |
| Deciders   | Jeremy (maintainer) |

## Context

`create-agent-system` ships Handlebars templates (`templates/`) and preset YAML files (`presets/`) as part of its npm package. At runtime, the tool must locate these bundled files to render agent definitions, skill files, and CLAUDE.md.

The challenge is that the resolution strategy must work across **all** of these scenarios simultaneously:

1. **Local development** -- `tsx src/index.ts` (unbundled, source TypeScript)
2. **After tsdown bundling** -- `node dist/index.js` (bundled ESM output)
3. **Global install** -- `npm install -g create-agent-system && create-agent-system`
4. **npx execution** -- `npx create-agent-system` (temporary install in npm cache)
5. **pnpm / yarn dlx** -- Similar to npx but different cache locations

The core question: **How does the bundled CLI find its own `templates/` and `presets/` directories regardless of where it is installed or how it is invoked?**

This is a well-known problem in the Node.js ecosystem. The answer differs fundamentally between CJS (`__dirname`) and ESM (`import.meta.url`) modules. Since this project uses ESM (as specified in the tech stack, bundled with tsdown per ADR-004), the strategy must be ESM-native.

## Decision

Use **`import.meta.url`** combined with **`fileURLToPath`** from the `node:url` module to resolve paths relative to the source file's location.

The resolution is implemented in a single utility:

```typescript
// src/utils/paths.ts
import { existsSync } from 'node:fs';
import { dirname, resolve } from 'node:path';
import { fileURLToPath } from 'node:url';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);

// Walk up from the current module's directory until we find package.json.
// This handles both cases:
//   - After tsup build: dist/index.js   → one level up = package root
//   - In development:   src/utils/paths.ts → two levels up = package root
function findPackageRoot(startDir: string): string {
  let dir = startDir;
  while (true) {
    if (existsSync(resolve(dir, 'package.json'))) return dir;
    const parent = dirname(dir);
    if (parent === dir) {
      throw new Error('Could not locate package root (no package.json found)');
    }
    dir = parent;
  }
}

const PACKAGE_ROOT = findPackageRoot(__dirname);

export const TEMPLATES_DIR = resolve(PACKAGE_ROOT, 'templates');
export const PRESETS_DIR = resolve(PACKAGE_ROOT, 'presets');
```

Key implementation details:

- **tsdown configuration** must set the output to `dist/index.js` (single level deep). The `../` relative path from `dist/` reaches the package root where `templates/` and `presets/` live.
- **package.json `files` field** must include `dist/`, `templates/`, and `presets/` so that npm publish packages the resource directories alongside the built code.
- **Development mode** uses a different depth (`../../` from `src/utils/`), so `PACKAGE_ROOT` computation uses the `findPackageRoot()` function that walks upward to find `package.json`, handling both depths automatically.
- **Note**: The spec's directory structure lists `src/utils/fs.ts`, `src/utils/detect.ts`, and `src/utils/constants.ts`. The `paths.ts` file is a new addition introduced by this ADR. Alternatively, the path resolution logic could be placed in `src/utils/constants.ts` alongside other path constants (`AGENTS_DIR`, `SKILLS_DIR`, etc.).

## Alternatives Considered

| Criterion              | import.meta.url + fileURLToPath        | __dirname (CJS)                        | process.cwd() + relative path           | Embed templates in code               |
|------------------------|----------------------------------------|----------------------------------------|-----------------------------------------|---------------------------------------|
| **ESM compatible**     | Yes (native ESM)                       | No (CJS only, `__dirname` is undefined in ESM) | Yes (but semantically wrong)    | Yes                                   |
| **Survives bundling**  | Yes (tsdown preserves `import.meta.url`) | N/A                                    | N/A                                     | Yes (templates become string literals)|
| **Global install**     | Works (resolves to install location)   | N/A                                    | Fails (resolves to user's cwd)          | Works                                 |
| **npx execution**      | Works (resolves to cache location)     | N/A                                    | Fails (resolves to user's cwd)          | Works                                 |
| **Template editing**   | Easy (edit .hbs files directly)        | N/A                                    | Easy                                    | Hard (edit string literals in TS)     |
| **Handlebars partials**| Supported (file-based)                 | N/A                                    | Supported                               | Complex (manual registration)         |

### Why not __dirname (CJS)?

This project uses ESM (`"type": "module"` in package.json, tsdown configured for ESM output). In ESM modules, `__dirname` is simply not available -- it is a CJS-only global. While there are shims (e.g., `createRequire`), they add complexity and fight against the module system rather than working with it. Using `import.meta.url` is the idiomatic ESM approach.

### Why not process.cwd() + relative path?

`process.cwd()` returns the **user's** current working directory, not the package's install location. When a user runs `npx create-agent-system` from `/home/user/my-project`, `process.cwd()` returns `/home/user/my-project`. The templates live in something like `/home/user/.npm/_npx/.../node_modules/create-agent-system/templates/`. These are completely different paths. This approach fundamentally cannot work for locating bundled resources.

### Why not embed templates in code?

Embedding templates as string literals (e.g., `const AGENT_TEMPLATE = "---\nname: ..."`) eliminates the path resolution problem entirely. However:

- **Handlebars loses its advantage** -- The entire point of using Handlebars is to keep templates as separate `.hbs` files that are readable and editable without touching TypeScript code. Embedding them defeats this purpose.
- **Partials and helpers become complex** -- Handlebars partials work naturally with file-based templates. With embedded strings, partial registration requires manual boilerplate.
- **Contributor experience degrades** -- Contributors who want to modify a template should edit a `.hbs` file, not hunt for a string literal inside a TypeScript module.
- **Readability** -- Multi-line template strings with YAML frontmatter and markdown inside TypeScript files are hard to read and maintain.

## Consequences

### Positive

- **Single idiomatic pattern** -- `import.meta.url` is the standard ESM way to get the current module's file path. No shims, no hacks, no polyfills.
- **Works everywhere** -- Global install, npx, pnpm dlx, local development -- all resolve correctly because `import.meta.url` always points to the actual file on disk.
- **tsdown compatibility** -- tsdown preserves `import.meta.url` in its ESM output by default. No special configuration needed for this specific feature.
- **Clean separation** -- Templates remain as `.hbs` files in their own directory. Handlebars can use file-system-based partials, helpers, and directory walking.
- **Debuggable** -- If path resolution fails, developers can log `import.meta.url` and `TEMPLATES_DIR` to immediately see where the tool is looking.

### Negative

- **Two-depth problem** -- In development (`src/utils/paths.ts`), the package root is `../../` away. After building (`dist/index.js`), it is `../` away. This is handled by the `findPackageRoot()` function that walks upward from the current module until it finds `package.json`. This adds a synchronous `existsSync` call at module load time, but it runs exactly once and the I/O cost is negligible for a CLI tool.
- **tsdown must not inline resources** -- If tsdown is ever misconfigured to bundle everything into a single file without preserving `import.meta.url`, the resolution breaks. The tsdown config needs a comment explaining why `import.meta.url` must be preserved.
- **`files` field maintenance** -- The `package.json` `files` array must be kept in sync. If someone adds a new resource directory (e.g., `schemas/`) but forgets to add it to `files`, it will not be published to npm. This is a standard npm concern, not specific to this approach, but worth noting.
