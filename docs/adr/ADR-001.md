# ADR-001: Argument Parser -- commander

| Field      | Value              |
|------------|--------------------|
| Status     | Accepted           |
| Date       | 2026-02-17         |
| Deciders   | Jeremy (maintainer) |

## Context

`create-agent-system` is a CLI tool that scaffolds Claude Code Agent Teams into projects. It needs an argument parser to handle:

1. **Positional arguments and flags** -- `--preset solo-dev`, `--project-name my-app`, `--yes`, `--no-run`
2. **Subcommands** -- The `validate` subcommand runs post-generation validation independently (e.g., `npx create-agent-system validate`)
3. **Non-interactive (CI) mode** -- All options must be passable as flags so the tool can run without prompts (`--yes` to skip confirmation)
4. **TypeScript compatibility** -- The entire codebase uses strict TypeScript; the parser must offer good type definitions

The spec lists two candidates: **citty** and **commander**. We need to choose one.

## Decision

Use **commander** (`commander` on npm) as the CLI argument parser.

Commander is installed as a runtime dependency and used in `src/cli/args.ts` to define the program, its options, and the `validate` subcommand.

```typescript
import { Command } from 'commander';

const program = new Command()
  .name('create-agent-system')
  .version('0.1.0')
  .option('--preset <name>', 'Preset to use (solo-dev | small-team | full-team)')
  .option('--project-name <name>', 'Project name')
  .option('--yes', 'Skip confirmation prompts')
  .option('--no-run', 'Do not launch Claude Code after scaffolding');

program
  .command('validate')
  .description('Validate an existing agent system configuration')
  .action(validateAction);
```

## Alternatives Considered

| Criterion           | commander                              | citty                                | yargs                                | minimist                    |
|---------------------|----------------------------------------|--------------------------------------|--------------------------------------|-----------------------------|
| **Subcommands**     | First-class `.command()` API           | Supported via `defineCommand` + `subCommands` | Supported via `.command()`   | No built-in support         |
| **TypeScript**      | Bundled `@types/commander` (DefinitelyTyped), widely tested | Native TS, excellent type inference | `@types/yargs` available, but complex generics | Minimal types               |
| **Bundle size**     | ~55 kB (acceptable for CLI)            | ~15 kB (very light)                  | ~150 kB+ (heavy)                     | ~6 kB                       |
| **Weekly downloads**| ~180M (npm top 20)                     | ~2M (growing but niche)              | ~100M                                | ~55M                        |
| **Ecosystem maturity** | 12+ years, battle-tested            | ~2 years, UnJS ecosystem             | 10+ years, stable                    | 10+ years, minimal updates  |
| **Docs & examples** | Extensive, large community             | Adequate, smaller community          | Extensive                            | Sparse                      |
| **Error messages**  | Helpful, auto-generated help/usage     | Good, auto-generated                 | Helpful                              | None (manual)               |

### Why not citty?

citty (from the UnJS ecosystem) is a strong modern alternative with native TypeScript support and a smaller footprint. However:

- **Smaller community** -- When contributors encounter issues with argument parsing, commander's extensive documentation, Stack Overflow presence, and GitHub issue history make debugging faster.
- **Subcommand pattern less established** -- While citty supports subcommands, the pattern is newer and has fewer real-world examples to reference. Commander's `.command()` API is one of the most documented CLI patterns in Node.js.
- **Risk vs. benefit** -- The 40 kB size difference is negligible for a CLI tool (not a library). The maturity and ecosystem advantages of commander outweigh citty's lighter footprint for this use case.

### Why not yargs?

yargs is mature and capable but significantly heavier (~150 kB+). For a scaffolding tool that needs simple flags and one subcommand, yargs introduces unnecessary complexity. Its middleware system and complex type generics add cognitive overhead without corresponding benefit.

### Why not minimist?

minimist is intentionally bare-bones. It provides raw argument parsing without help generation, subcommand support, or validation. We would need to build all of that manually, which contradicts the "simple is better" principle -- simple means using a tool that already solves the problem, not reimplementing it.

## Consequences

### Positive

- **Immediate productivity** -- Commander's API is familiar to most Node.js developers; onboarding friction is near zero.
- **Subcommand support out of the box** -- The `validate` subcommand (and any future subcommands like `add` in v0.3.0) work with a clean, well-documented API.
- **Auto-generated help** -- Running `npx create-agent-system --help` produces formatted help text without manual effort.
- **Proven reliability** -- At ~180M weekly downloads, edge cases have been found and fixed. We are unlikely to encounter parser bugs.
- **Composability with clack** -- Commander handles flag parsing, then hands off to clack for interactive prompts. The two libraries have clean separation of concerns.

### Negative

- **Not the lightest option** -- At ~55 kB, commander is heavier than citty or minimist. This is a non-issue for a CLI tool but worth noting.
- **CJS-first history** -- Commander was originally a CJS library. While it fully supports ESM (and has for years), some older examples online show CJS patterns, which can confuse contributors.
- **External types** -- Unlike citty's native TypeScript, commander's types come from DefinitelyTyped. In rare cases, types may lag behind a new release. In practice, this has not been an issue for years.
