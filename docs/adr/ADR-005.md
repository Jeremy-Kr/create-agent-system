# ADR-005: Template Engine -- Handlebars

| Field      | Value              |
|------------|--------------------|
| Status     | Accepted           |
| Date       | 2026-02-17         |
| Deciders   | Jeremy (maintainer) |

## Context

`create-agent-system` generates markdown files (agent definitions, skill files, CLAUDE.md) from templates. Each template contains:

- **YAML frontmatter** with dynamic values (agent name, model, skills list)
- **Markdown body** with conditional sections (e.g., Visual QA instructions only when enabled)
- **Loops** for rendering lists (e.g., active agents table, file ownership table)
- **Variable injection** for project-specific values (project name, package manager, tech stack)

The template engine must:

1. **Handle markdown naturally** -- Templates are primarily markdown with embedded directives. The engine should not fight markdown syntax.
2. **Support conditionals and loops** -- `{{#if visualQa}}`, `{{#each activeAgents}}` patterns are needed.
3. **Support partials** -- Reusable template fragments (e.g., a shared "References" section across agent templates).
4. **Keep templates readable** -- Contributors editing templates should understand them without deep JavaScript knowledge.
5. **Work with file-based templates** -- Templates live in `templates/` as separate files (see ADR-002), not embedded in code.

The spec lists Handlebars as the template engine. This ADR documents the rationale and alternatives analysis.

## Decision

Use **Handlebars** (`handlebars` on npm, v4.7.8) as the template engine.

Templates are stored as `.hbs` files in the `templates/` directory and rendered by the template renderer (`src/core/template-renderer.ts`):

```typescript
import Handlebars from 'handlebars';
import { readFileSync } from 'node:fs';

export function renderTemplate(templatePath: string, data: Record<string, unknown>): string {
  const source = readFileSync(templatePath, 'utf-8');
  const template = Handlebars.compile(source);
  return template(data);
}
```

Handlebars' logic-less philosophy is a deliberate choice: templates cannot contain arbitrary JavaScript. This constrains what template authors can do, which is a feature for this project -- templates should be declarative descriptions of output structure, not programs.

### Why not EJS?

EJS allows arbitrary JavaScript in templates (`<% for (let i = 0; i < items.length; i++) { %>`). For a scaffolding tool where multiple contributors may edit templates, this is a liability:

- **Template complexity creep** -- Nothing prevents a contributor from adding complex data transformations, API calls, or error handling inside templates. Handlebars' logic-less approach makes this structurally impossible.
- **Security surface** -- EJS templates can execute arbitrary code. While this project's templates are bundled (not user-provided), the principle of least authority still applies.
- **`<% %>` delimiters** -- EJS delimiters can visually clash with HTML-like content. For markdown templates with YAML frontmatter, `{{ }}` delimiters (Handlebars/Mustache) read more naturally.

### Why not Eta?

Eta is the strongest modern alternative. It is TypeScript-native, ~9x smaller than Handlebars, and ~7x faster in benchmarks. However:

- **Full JS in templates** -- Like EJS, Eta allows arbitrary JavaScript in templates. The same template complexity concerns apply.
- **Smaller community** -- At ~2M weekly downloads vs Handlebars' ~29M, fewer contributors will be familiar with Eta's syntax and conventions.
- **Benchmarks are irrelevant here** -- This is a scaffolding CLI that renders ~15 templates once per invocation. The total rendering time is under 10ms regardless of engine choice. Eta's speed advantage provides no practical benefit.
- **Migration cost if needed later is low** -- If Handlebars becomes genuinely unmaintainable (security vulnerability with no patch), migrating to Eta is straightforward: the template syntax differences are small, and the rendering API is similar.

### Why not Mustache?

Mustache is Handlebars' predecessor and shares the `{{ }}` syntax. However, Handlebars is a strict superset of Mustache with critical additions:

- **`{{#if}}` / `{{else}}`** -- Mustache's `{{#section}}` only tests for truthiness; it cannot express `if/else` branches cleanly. Handlebars' `{{#if condition}}...{{else}}...{{/if}}` pattern is needed for conditional template sections.
- **`{{#each}}` with `@index`, `@key`** -- Mustache iterates with `{{#list}}...{{/list}}` but lacks loop metadata. Handlebars provides `@index`, `@first`, `@last` which are useful for table rendering.
- **Custom helpers** -- Handlebars supports custom helpers (e.g., `{{join skills ", "}}` for comma-separated skill lists). Mustache has no helper mechanism.
- **Performance** -- Handlebars renders 5-7x faster than Mustache due to precompilation.

### Why not tagged template literals?

Using ES6 tagged template literals eliminates the template engine dependency entirely:

```typescript
const agentTemplate = (data) => `---
name: ${data.name}
description: "${data.description}"
---
# Agent: ${data.name}
${data.visualQa ? '## Visual QA\n...' : ''}
`;
```

However:

- **Readability collapses** -- Multi-line markdown templates with YAML frontmatter, conditional blocks, and loops become unreadable as string literals. Escaping backticks (common in markdown code blocks) adds further noise.
- **No file-based templates** -- Templates must live inside TypeScript files. Contributors who want to edit a template must navigate TypeScript code rather than opening a `.hbs` file. This contradicts the ADR-002 decision to keep templates as separate files.
- **No partials** -- Reusable template fragments require manual function extraction and composition, losing the declarative `{{> partial}}` pattern.
- **No separation of concerns** -- Template logic and application logic merge. Changes to template content require TypeScript compilation and linting, even if only markdown text changed.

## Consequences

### Positive

- **Template safety** -- Handlebars' logic-less philosophy prevents arbitrary code execution in templates. Template authors can focus on structure and content without the temptation (or risk) of embedding complex logic.
- **Familiar syntax** -- `{{ }}` is the most recognized template syntax in the JavaScript ecosystem. Contributors from any background (Ruby/ERB, Python/Jinja2, Go/text.template) will recognize the patterns.
- **Partials for DRY templates** -- Shared sections (e.g., scoring references, file ownership preambles) can be extracted into partials and reused across agent templates without duplication.
- **Markdown-friendly** -- `{{ }}` delimiters do not clash with markdown syntax (unlike `<% %>` which can conflict with HTML blocks). YAML frontmatter and markdown body coexist naturally.
- **Proven at scale** -- At ~29M weekly downloads and 12+ years of existence, Handlebars' edge cases are well-documented. The template syntax is stable and unlikely to change.

### Negative

- **Maintenance concern** -- Handlebars v4.7.8 was released 3+ years ago with no subsequent releases. While this reflects stability (not abandonment -- the library is feature-complete for most use cases), it means security vulnerabilities would be slow to patch. **Mitigation**: Handlebars is used at build time only (scaffolding), not at runtime in end-user applications. The attack surface is limited to the CLI developer's machine.
- **Larger bundle than alternatives** -- At ~22 kB gzipped, Handlebars is larger than Eta (~2.5 kB) or Mustache (~4 kB). For a CLI tool, this is negligible (adds ~0.01s to cold start), but worth noting.
- **External types** -- Handlebars types come from DefinitelyTyped (`@types/handlebars`), not the library itself. Type accuracy may lag behind features, though this has not been an issue with the stable v4.7.x API.
- **No native TypeScript** -- Unlike Eta, Handlebars is written in JavaScript. For a TypeScript-strict project, the type experience is adequate but not native. Helper registration and template data typing require manual type annotations.
