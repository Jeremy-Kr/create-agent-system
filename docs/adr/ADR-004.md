# ADR-004: Build Tool -- tsdown (replacing tsup)

| Field      | Value              |
|------------|--------------------|
| Status     | Accepted           |
| Date       | 2026-02-17         |
| Deciders   | Jeremy (maintainer) |
| Supersedes | Spec tech stack (tsup) |

## Context

The spec lists **tsup** as the build tool for bundling the CLI into distributable ESM output. However, tsup is no longer actively maintained -- its maintainer recommends **tsdown** as the successor.

`create-agent-system` needs a build tool that:

1. **Bundles TypeScript to ESM** -- The CLI is distributed as a single ESM entry point (`dist/index.js`)
2. **Generates type declarations** -- Consumers of the library types (if any) need `.d.ts` files
3. **Preserves `import.meta.url`** -- Critical for locating bundled `templates/` and `presets/` directories at runtime (see ADR-002)
4. **Is actively maintained** -- Security patches, Node.js version compatibility, and ecosystem evolution require ongoing maintenance

tsup (built on esbuild) served the ecosystem well, but its maintenance has stalled. tsdown (built on Rolldown, a Rust-based bundler) is the recommended migration path with near-identical configuration and automated migration tooling.

## Decision

Use **tsdown** (`tsdown` on npm, currently v0.20.3) as the build tool, replacing tsup from the original spec.

```typescript
// tsdown.config.ts
import { defineConfig } from 'tsdown';

export default defineConfig({
  entry: ['src/index.ts'],
  format: 'esm',
  dts: true,
  clean: true,
});
```

Key differences from the spec's tsup configuration:

- **Config file**: `tsdown.config.ts` instead of `tsup.config.ts`
- **`format` defaults to `esm`** -- Aligns with this project's ESM-only target. No need to explicitly set it, but we do for clarity.
- **`clean` is enabled by default** -- tsdown wipes `outDir` before builds. This matches tsup's `clean: true` behavior.
- **`dts` auto-detection** -- tsdown automatically enables DTS generation if `package.json` contains `types` or `typings` fields. We set it explicitly for clarity.
- **`shims` option removed** -- The spec's tsup config included `shims: true` for `__dirname` compatibility. tsdown does not have this option. This is intentional: the project uses `import.meta.url` (see ADR-002), so `__dirname` shims are unnecessary.

### Migration Path

tsdown provides an automated migration CLI:

```bash
npx tsdown-migrate
```

This converts `tsup.config.ts` to `tsdown.config.ts` format. Since this is a greenfield project (no existing tsup config to migrate), we configure tsdown directly.

## Alternatives Considered

| Criterion             | tsdown (chosen)                        | tsup                                   | Rollup + plugins                       | esbuild (direct)                       |
|-----------------------|----------------------------------------|----------------------------------------|----------------------------------------|----------------------------------------|
| **Actively maintained** | Yes (v0.20.3, Feb 2026, 77 contributors) | No (maintainer recommends tsdown)     | Yes (but complex for library bundling) | Yes (but minimal DTS support)          |
| **Engine**            | Rolldown (Rust)                        | esbuild (Go)                           | Rollup (JS + Rust via rolldown)        | esbuild (Go)                           |
| **DTS generation**    | Built-in via Oxc (fast)                | Built-in via rollup-plugin-dts (slower)| Manual plugin setup                    | Not supported                          |
| **ESM output**        | Default                                | Supported                              | Native                                 | Supported                              |
| **Config compat**     | tsup-compatible options                | N/A (original)                         | Different config format                | Different config format                |
| **`import.meta.url`** | Preserved in ESM output                | Preserved in ESM output                | Preserved                              | Preserved                              |
| **Build speed**       | Very fast (Rust-based)                 | Fast (Go-based)                        | Moderate                               | Very fast                              |
| **Ecosystem**         | 3.4k stars, 4.5k dependents           | ~34k stars, widely used (legacy)       | ~25k stars, mature                     | ~38k stars, low-level                  |

### Why not keep tsup?

tsup is no longer actively maintained. While it still works, relying on an unmaintained build tool means:

- **No security patches** -- If a vulnerability is found in tsup or its esbuild dependency handling, fixes may not come.
- **No Node.js compatibility updates** -- As Node.js evolves (v22, v24), tsup may not keep up.
- **Ecosystem divergence** -- The Vite ecosystem is converging on Rolldown (Vite 8 will use Rolldown). tsdown aligns with this trajectory; tsup does not.

### Why not Rollup + plugins?

Rollup is powerful but requires manual plugin configuration for TypeScript, DTS generation, and ESM output. For a single-entry CLI tool, this introduces unnecessary configuration complexity. tsdown provides the same Rolldown engine with sensible defaults.

### Why not esbuild directly?

esbuild is fast but does not generate TypeScript declaration files. A separate `tsc --emitDeclarationOnly` step would be needed, adding build complexity. tsdown handles both bundling and DTS in a single command.

## Consequences

### Positive

- **Active maintenance** -- tsdown is under active development (166 releases, 77 contributors as of Feb 2026). Security patches and Node.js compatibility updates will continue.
- **Faster DTS generation** -- tsdown uses Oxc for type declaration generation, which is significantly faster than tsup's rollup-plugin-dts approach. This matters as the project grows.
- **Future-aligned** -- Rolldown is becoming the foundation of the Vite ecosystem (Vite 8). Building on tsdown means the project benefits from this investment.
- **Near-identical DX** -- tsdown's CLI and config format are deliberately compatible with tsup. Contributors familiar with tsup will feel at home.
- **ESM-first defaults** -- tsdown defaults to ESM output, matching this project's module strategy without requiring explicit configuration.

### Negative

- **Smaller community (currently)** -- At 3.4k stars, tsdown has a smaller community than tsup's 34k. Debugging help via Stack Overflow is more limited. However, the community is growing rapidly and GitHub issues are actively triaged.
- **Feature gaps** -- Some tsup features are not yet implemented in tsdown. The tsdown docs explicitly note this. For this project's simple configuration (single entry, ESM, DTS), no missing features are expected, but edge cases may arise.
- **Spec deviation** -- The original spec lists tsup. This ADR formally supersedes that choice. All tickets referencing tsup (e.g., TICKET-001's `tsup.config.ts`) must be updated to reference tsdown.
